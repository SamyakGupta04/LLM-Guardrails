    sees "ignore" as a BYPASS_TERM and "policy" as a TARGET_TERM, triggering
    proximity_bypass. The ICD (Imperative Command Density) score is 1.0 because
    "do not ignore" is parsed as imperative. Combined score = 0.80.
    SEVERITY: HIGH - Insurance customers regularly use "ignore" in legitimate
    contexts (e.g., "don't ignore the deadline").
    FIX: The proximity_hit function should check for NEGATED bypass terms.
    "do not ignore" should be treated differently from "ignore".

11. JSON SCANNER  (10/13 passed = 76.9%)
------------------------------------------------------------------------

  WHAT WORKS WELL:
  - Valid JSON (simple, nested, in text): Correctly allowed
  - Invalid JSON with repair=False: Correctly flagged
  - Invalid JSON with repair=True: Successfully repaired
  - No JSON in text: Correctly handled
  - Required elements validation: Works correctly
  - Deeply nested JSON: Handled correctly
  - Empty text: Handled correctly

  WHAT WENT WRONG:
  [FALSE NEGATIVE] '{name: "John", age: 30}' - Not flagged as invalid
  [FALSE NEGATIVE] '{"name": "John", "age": 30,}' - Not flagged as invalid
  [FALSE NEGATIVE] Malformed JSON in long text - Not flagged

    ROOT CAUSE: The JSON scanner uses repair=True by DEFAULT. When repair
    succeeds, the JSON is considered "valid" and no issue is raised. So:
    - '{name: "John"}' -> json_repair fixes missing quotes -> valid -> no issue
    - '{"name": "John",}' -> json_repair removes trailing comma -> valid -> no issue

    This is CORRECT BEHAVIOR for the default repair=True mode. The tests
    expected issues to be raised even when repair succeeded, which is wrong.
    The scanner's logic: "If I can fix it, it's fine."

    SEVERITY: LOW - This is actually a test expectation error, not a bug.
    The json_repair library is working as designed. If you want to flag
    repaired JSON as an issue, that would require a design change.

    NOTE: The "malformed_json_in_long_text" test also repairs successfully
    because json_repair handles the {name: "John"} pattern well.

12. BAN TOPICS SCANNER  (8/12 passed = 66.7%)  *** CRITICAL ***
------------------------------------------------------------------------

  WHAT WORKS WELL:
  - Violence, hate speech, sexual content, political propaganda,
    criminal instructions: All correctly detected
  - Keyword-boosted detection (kill, murder, slaughter): Working
  - Long text with hidden banned content: Detected
  - Empty text: Handled correctly

  WHAT WENT WRONG:
  [FALSE POSITIVE] Safe insurance response flagged as "criticism of government"
    SCORE: 0.63 > threshold 0.60
  [FALSE POSITIVE] Financial advice flagged as "encouraging violence"
    SCORE: 0.67 > threshold 0.60
  [FALSE POSITIVE] Customer service response flagged as "criticism of government"
    SCORE: 0.73 > threshold 0.60
  [FALSE POSITIVE] Neutral crime reporting flagged as "encouraging violence"
    SCORE: 0.70 > threshold 0.60

    ROOT CAUSE: The DEFAULT_THRESHOLD of 0.60 is too LOW for the MiniLM
    embedding model. The semantic similarity between benign content and
    banned intent descriptions is inherently above 0.6 for many normal texts.

    "Thank you for contacting us. Your claim has been processed..." scores
    0.73 against "criticism of government" - this is clearly a model
    limitation, not a content issue.

    The MiniLM model's 384-dimensional embeddings produce relatively high
    cosine similarities between any two English sentences of similar length/
    structure, regardless of actual semantic overlap.

    SEVERITY: CRITICAL - In production, this would block legitimate insurance
    responses, financial advice, and customer service messages. This makes
    the ban_topics scanner unusable in its current form.

    FIX OPTIONS:
    1. Raise DEFAULT_THRESHOLD to 0.75-0.80 (reduces false positives but
       may miss subtle banned content)
    2. Use a cross-encoder model instead of bi-encoder for more accurate
       similarity measurement
    3. Add a "safe content" allowlist embedding set and require that banned
       score > safe score by a margin
    4. Implement a two-stage approach: semantic pre-filter + LLM classifier
       for borderline cases
    5. Use more specific intent descriptions instead of broad categories
